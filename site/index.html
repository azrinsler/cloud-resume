Networking

In order to access any website, you need to - at some level - know its IP address. Asking users to remember this address would of course be completely unreasonable, which is where DNS comes into play. DNS is what maps between IP addresses and human-readable domain names.
 
By default, Amazon makes it relatively quick and easy to statically host a website using S3; you simply upload your files to a bucket marked for that purpose, make sure it's publicly readable, and Amazon generates a URL for you. The URL it generates is based on the name of the bucket, so you end up with something like http://azrinsler-site-bucket.s3-website-us-east-1.amazonaws.com/.
 
While certainly better than a "raw" IP address, there are still several problems with serving your content through these. The foremost, as you may have noticed, being that the URL is http instead of https - a security flaw. One serious enough that most browsers will warn users away from even accessing the site in the first place! A secondary issue with this URL is that it is kind of long and difficult to remember. We can address these issues using a combination of Route 53, ACM (AWS Certificate Manager), and CloudFront.


 

ACM is the simplest and most straightforward of these three services, in that "all" it's doing is providing us with the SSL certificate(s) we need in order to enable HTTPS. Each certificate we make in ACM is for a specific root domain, and contains one or more CNAME (Canonical Name) DNS records related to that domain. CNAME records by themselves don't do much besides map between each other, and these particular CNAME's point to Amazon-owned records related to the validation of our certificate. You need one record for the root domain name, and may need more for any subdomains based on that root. The first and most obvious subdomain we will need is the "www" one. It would also be convenient to have a subdomain for API access, so we create one for "api" as well. Note that this certificate does not perform any routing, etc. for us by itself, nor does it actually register the domain name we want it to use.


 

Amazon - through Route 53 - is a domain registrar. So we can use them to register the domain name we want to use along with its corresponding TLD (top-level domain). The "com" TLD is both the cheapest and most common option, and costs about $15/year to register through Amazon.
 
Route 53 also provides us with the ability to additional DNS records of different types, which we will need to do if we want our website to actually be found when somebody attempts to navigate to it. In order to define these additional records, we need to first create a hosted zone within Route 53. Hosted zones, by default, come with a few records of their own. This includes one for NS (name server) and one for SOA (start of authority). You can read more about these here.
 
In addition to these, we will need a number of "A" records - these are how we connect our domain to its IPv4 address. If we wanted to use IPv6, these would instead need to be "AAAA" records. We could have these point directly to our site content in S3, but that by itself would not solve for the original, primary issue with S3 (the lack of HTTPS).


 

For this, we need to use CloudFront. CloudFront provides a variety of different services, but right now the one we're most interested in is the ability to redirect HTTP traffic to HTTPS. We can accomplish this by creating a new CloudFront distribution, and adding our S3 bucket as an origin. Redirecting HTTP to HTTPS is one of the options we can select for its behavior. Easy enough!
 
Much like the S3 bucket itself, Amazon gives our CloudFront distribution its own programmatically generated URL. Something like https://dpqmp8uwwgxwd.cloudfront.net which is... unfortunately difficult to read, let alone remember. We can use Route 53 to solve this problem; in order to do so we must first update our CloudFront distribution to use the same name server(s) as Route 53, which we can easily check using our NS record.


 

Now that we have a CloudFront distribution set up, we can finish defining records in Route 53. To start with, we'll need "A" records which connect our registered domain (and subdomain) names to whatever URL AWS gave to it. Since CloudFront is set to redirect HTTP to HTTPS, we will also need to include our CNAME records from ACM. While these "A" and CNAME records serve different purposes, for our case we can treat them as pairs:


 
  • One pair for the root domain
  • One pair for the "www" subdomain
  • One pair for the "api" subdomain

 

With all of this set up, we should finally be able to access our website in S3 using HTTPS and at a custom domain name of our choosing! That takes care of our root domain, as well as the "www" subdomain. All that's left is to connect our "api" subdomain to an actual API.


 

In order to create an API through AWS, we'll use API Gateway. The actual specifics of how it works are more of a backend implementation detail, but thankfully we can worry about that at a different time. For now; we can simply create a custom domain (within API Gateway) using our subdomain name, and update our "api" "A" record in Route 53 to point to it. Much like with CloudFront, the default URL assigned to our custom domain in API gateway is largely unintelligible to a human reader, so this record is the difference between users accessing your API at something like api.azrinsler.com vs. d-37rxfs2ecj.execute-api.us-east-1.amazonaws.com.
 
This custom domain is internally connected to a specific stage within an API Gateway API, but as mentioned that is more of an implementation detail. Each API implemented in this way will have its own generated URL in addition to that of the custom domain, so we can add that one to the list of unintelligible URLs we could technically use, but prefer to avoid due to readability issues. For instance, the API stage used by this site is located at https://gms9t124k8.execute-api.us-east-1.amazonaws.com/.

Following all this setup, we have the following:


 
  1. Our primary domain, located at azrinsler.com
    • This works with or without the "www" prefix
    • It automatically redirects HTTP traffic to HTTPS
    • Traffic going through CloudFront internally uses dpqmp8uwwgxwd.cloudfront.net
    • The S3 bucket the site originates from is located at azrinsler-site-bucket.s3-website-us-east-1.amazonaws.com

     
  2. An API located at api.azrinsler.com
    • When going through CloudFront, this only accepts HTTPS traffic
    • It points to a custom domain at d-37rxfs2ecj.execute-api.us-east-1.amazonaws.com
    • The custom domain itself internally uses gms9t124k8.execute-api.us-east-1.amazonaws.com/

Devops

Build automation, DevOps, Lean, CI/CD, IaC (Infrastructure as Code)... these terms are often used somewhat interchangeably. They are highly related concepts which - broadly speaking - focus on the speed, quality, and consistency of your build and development processes. Much like QE, DevOps is frequently overlooked or deprioritized in favor of more visible and pressing aspects of software development, with the consequences of doing so only becoming apparent at a late stage. Why is that?


 

There are of course a myriad of possible reasons, but I would argue that most of them fall into one of these categories:

  • DevOps is its own skill set involving a different tech stack, which developers are not necessarily familiar with.
  • Automation in general is heavily front-loaded in terms of effort, which makes it painful to set up.
  • The benefits are frequently invisible - it's hard to appreciate the value a seamless deployment if you haven't experienced the pain of a botched one.

 

The fact is; having a highly automated build process is of absolutely foundational importance to all but the smallest of projects.


 

This project leverages the following:

  • GitHub Actions - Build Automation
  • GitHub Packages - Artifact Storage
  • Maven - Dependency Management
  • Terraform - IaC

 

By using these tools together, we can create a cohesive build process which quietly handles many of the common pain points surrounding CI/CD. GitHub Actions and Packages are of course tightly coupled to GitHub itself, which is where the codebase already resides. By using these we can avoid the additional setup and maintenance involved in alternative tools, such as Jenkins or JFrog/Artifactory.
 
Maven greatly simplifies dependency management in addition to being a powerful build tool in its own right. A custom settings.xml file allows Maven to retrieve dependencies from GitHub packages without exposing credentials. I could have just as easily used Gradle for this same purpose, and mostly went with Maven out of personal preference.
 
Being that this is an AWS-based project, Cloudformation would typically be the go-to option for IaC. So why Terraform? Well, mostly because Terraform is open-source and vendor-agnostic. In order to avoid any additional cost or dependency on Hashicorp themselves, this project stores its state in S3.
